`docs/refactor-logs/20260425-serverless-lanes/03_composer2_base_plan.md` の基本計画を精読し、指定された制限事項に基づき監査しました。

### 1. 事実誤認の指摘

*   **ライブラリ依存関係の誤認:** 基本計画では `runpod` および `vastai` クライアントライブラリの抽象化が提案されていますが、これらのスクリプトは「プロトタイプ・プローブ（疎通確認スクリプト）」であり、`src/gpu_job` 下のプロダクションコード（Provider Adapter）とは意図的に疎結合に保たれています。プロダクション側の共通基盤に依存させる変更は、プローブとしての独立性を損なうため事実上の設計ミスです。
*   **例外処理の共通化範囲:** 両スクリプトで共通の `ContractError` クラスを導入する案がありますが、現状のスクリプトは個別の API レスポンス形式（RunPod の `status` と Vast.ai の `state`）に特化した例外判定を行っており、共通化すると判定精度が低下します。

### 2. 過剰なリファクタ案の却下

以下の項目は「最小安全」の原則に反し、Provider Adapter との境界を曖昧にするリスクがあるため却下します。

*   **却下:** `BaseProbe` 基底クラスの導入。
    *   理由: 継承構造を導入すると、一方のプロバイダーの API 変更が他方のスクリプトの実行に影響を与える恐れがあります。
*   **却下:** `pydantic` モデルへの完全移行。
    *   理由: 依存ライブラリの更新を強いることになり、CI 環境（`all green`）の破壊リスクが高まります。現状の辞書ベースのバリデーションでプローブとしては十分です。
*   **却下:** ログ出力の `logging` モジュールへの統一。
    *   理由: `scripts/` 下のツールは `print` による標準出力が後続のシェルパイプライン（`jq` 等）で利用されることが前提となっており、副作用が大きすぎます。

### 3. 最小安全候補と優先順位

「All Green」を維持しつつ、保守性を高めるための最小変更のみを残します。

| 優先順位 | 変更内容 | 対象 | 効果 |
| :--- | :--- | :--- | :--- |
| **1 (高)** | タイムアウト値の定数化 | 両スクリプト | ネットワーク不安定によるテスト失敗（Flaky）の防止 |
| **2 (中)** | API レスポンスのキー存在チェックの厳格化 | `verify_contract` 関数 | 異常終了時のエラーメッセージの具体化 |
| **3 (低)** | docstring のフォーマット統一 | ファイル冒頭 | 自動ドキュメント生成の整合性確保 |
